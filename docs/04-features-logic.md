# Бизнес-логика

Этот файл описывает вычисления и правила поведения, общие для web и mobile клиентов. Реализующий должен стремиться вынести эту логику в `/packages/shared` (чистые функции без зависимости от React/React Native), чтобы не дублировать её отдельно в двух кодовых базах — но если по времени/сложности это неудобно, дублирование в двух местах с одинаковой логикой также приемлемо.

## Расчёт streak (дней подряд)

**Определение:** streak — это количество последовательных дней (включая сегодня), в каждый из которых пользователь выполнил хотя бы одну задачу.

**Источник данных:** таблица `daily_history(user_id, day, tasks_completed)`.

**Алгоритм:**
```
streak = 0
currentDate = сегодня (по локальной таймзоне пользователя)
while true:
  найти строку daily_history где day = currentDate
  если строка существует и tasks_completed > 0:
    streak += 1
    currentDate = currentDate - 1 день
  иначе:
    прервать цикл
return streak
```

**Важный нюанс:** если пользователь ещё не отметил ни одной задачи сегодня, но вчера отмечал — streak не должен прерываться "досрочно" в течение текущего дня. То есть: если сегодняшней строки в `daily_history` ещё нет (`tasks_completed` отсутствует или 0), цикл просто не считает сегодня, но продолжает считать от вчера. Практически это означает: начинать проверку с сегодня, и если сегодня пусто — пропустить день и продолжить с вчера для базового отображения "текущего" streak, либо явно показывать пользователю "у тебя был streak N дней, отметь что-то сегодня, чтобы продолжить" — конкретная UX-формулировка на решение реализующего, но логика подсчёта должна не наказывать пользователя за то, что день ещё не закончился.

Рекомендация для простоты реализации (MVP-вариант): считать streak только по полностью прошедшим дням + сегодня, если на сегодня уже что-то отмечено. Если на сегодня пока пусто — показывать streak за вчера и более ранние дни как "текущий", без обнуления.

## Расчёт "задач за неделю"

Сумма `tasks_completed` из `daily_history` за последние 7 дней (включая сегодня), скользящее окно, не привязанное к календарным неделям (не "с понедельника", а именно последние 7 календарных дней от текущего момента).

```
weekTotal = sum(tasks_completed) where day >= (сегодня - 6 дней) and day <= сегодня
```

## Отметка задачи выполненной

При переключении чекбокса задачи с `false` на `true`:
1. `update tasks set done=true, done_at=now() where id=...`
2. Upsert в `daily_history`: если строка для `(user_id, day=today)` существует — `tasks_completed = tasks_completed + 1`, иначе создать новую строку с `tasks_completed=1`.

При обратном переключении (с `true` на `false`, если пользователь случайно отметил и хочет снять отметку):
1. `update tasks set done=false, done_at=null where id=...`
2. Декремент `tasks_completed` в `daily_history` за тот день, когда задача была изначально отмечена (не за сегодня, если задача была отмечена в другой день — хотя в этой модели задачи привязаны к конкретному дню `day`, так что это обычно один и тот же день). Не дать `tasks_completed` уйти в отрицательное значение — `max(0, tasks_completed - 1)`.

## Ежедневное создание дефолтных задач

При открытии приложения (на любом клиенте) — проверить, есть ли в таблице `tasks` хотя бы одна строка с `day = сегодня` для текущего `user_id`. Если нет — создать 3 дефолтные задачи (см. список в `01-data-model.md`, раздел "Сиды", пункт 3). Это гарантирует, что каждый новый день у пользователя есть стартовый набор задач, а не пустой экран.

Эта проверка должна выполняться один раз при инициализации приложения/экрана "Сегодня", не на каждый рендер.

## Прогресс-бары обучения

Простое отношение `hours / goal_hours * 100`, ограниченное сверху 100% (`Math.min(100, ...)`), округлённое до целого числа для отображения. Значения `hours` и `goal_hours` берутся напрямую из таблицы `study_hours`.

Кнопки `+1 час` / `-1 час` обновляют `hours` напрямую (`update study_hours set hours = hours + 1 where user_id=... and track=...`), не давая уйти в отрицательные значения (`max(0, hours - 1)` на клиенте перед отправкой, либо `check` constraint в БД).

## Расчёт "занятых часов" в недельном графике

Чисто информативное число, показывающее пользователю примерную загрузку недели по графику (не привязано к реальным отметкам выполнения, просто оценка по шаблону):

```
занятыеЧасы = (количество слотов с slot_type не равным 'empty' и не равным 'rest') * 2.5
```

Множитель 2.5 (часа на блок) — приближение по умолчанию, обозначающее, что один блок (утро/день/вечер) занимает примерно 2-3 часа активной работы. Это не точное значение, в UI нужно явно подписать как «≈» (примерно), а не выдавать за точный расчёт.

## Статусы откликов — нет автоматических переходов

В отличие от streak/прогресс-баров, статус отклика (`applied → screen → interview → offer/rejected`) НЕ должен автоматически продвигаться по таймеру или дате — это вручную обновляемое поле пользователем через select на экране "Отклики". Единственная автоматика — `updated_at` обновляется на `now()` при любом изменении статуса.

## Часовые пояса и даты

Все поля `day` (date, без времени) должны интерпретироваться в **локальной таймзоне устройства пользователя**, не в UTC сервера. Это важно, чтобы "сегодня" на клиенте совпадало с "сегодня" в базе данных независимо от того, в каком часовом поясе физически находится Supabase-сервер. На клиенте при формировании значения `day` для записи использовать локальную дату (`new Date().toLocaleDateString()` с последующим приведением к формату `YYYY-MM-DD`, либо библиотеку типа `date-fns` с явной работой в локальной таймзоне), не `new Date().toISOString()` (которая сконвертирует в UTC и может "перепрыгнуть" дату вечером).
