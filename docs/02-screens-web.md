# Экраны web-приложения (Next.js)

## Навигация

Верхний уровень — горизонтальные табы (как во вкладках браузера), без бокового меню — приложение неглубокое, 5 разделов:

```
[Сегодня] [Недельный график] [Отклики] [Прогресс] [Ресурсы и ссылки]
```

Роутинг через App Router: каждый таб — отдельный route segment, чтобы был прямой URL и кнопка "назад" в браузере работала ожидаемо.

```
/today           — чек-лист на сегодня (главная / дефолтная страница после логина)
/schedule        — недельный график
/applications    — трекер откликов
/progress        — прогресс по обучению и статистика
/resources       — справочник ссылок (статический контент, без БД)
/login           — форма входа/регистрации
```

## Общий layout

- Шапка: имя пользователя/email + три счётчика-карточки (streak дней подряд, задач за неделю, всего откликов) — те же данные, что в исходном HTML-прототипе.
- Тёмная тема по умолчанию, токены см. в `00-overview.md`.
- Состояние авторизации проверяется в layout — если пользователь не залогинен, редирект на `/login`.

## Экран `/today` — Сегодня

**Назначение:** ежедневный чек-лист + быстрая рефлексия. Это экран, который пользователь открывает каждый день первым.

**Данные:** `tasks` где `day = текущая дата` (по локальному времени пользователя), отсортированные по `created_at`.

**Компоненты:**
- Список задач с чекбоксами. Каждая строка: чекбокс, текст задачи, цветной бейдж категории (job/study/eng), всё в одну строку. Выполненные задачи — текст с зачёркиванием и приглушённым цветом.
- При клике на чекбокс: optimistic update в UI + `update tasks set done=true, done_at=now() where id=...` + upsert в `daily_history` (инкремент `tasks_completed`).
- Форма добавления новой задачи внизу списка: текстовое поле + select категории (job/study/eng) + кнопка "Добавить". При сабмите — insert в `tasks` с `day` = сегодня.
- Отдельный блок ниже — текстовое поле "Заметка дня / рефлексия" (textarea), с кнопкой "Сохранить" или сохранением по `onBlur`. Данные из/в таблицу `daily_notes` (upsert по `user_id + day`).

**Пустое состояние:** если на сегодня нет задач (например, пользователь их все удалил) — показать текст-приглашение добавить первую задачу, без графики, просто текстом по центру.

## Экран `/schedule` — Недельный график

**Назначение:** настраиваемый шаблон недели — какие блоки времени на что отведены.

**Данные:** `schedule_slots` (21 строка на пользователя).

**Компоненты:**
- Поле "часов в неделю" (number input) — это не отдельное поле в БД, а просто ориентир для пользователя; можно хранить как пользовательскую настройку в отдельной маленькой таблице `user_settings(user_id, hours_per_week)` либо просто в localStorage/AsyncStorage клиента, так как это не критичные для синхронизации данные.
- Сетка 8 колонок × 4 строки: первая колонка — подписи блоков (Утро/День/Вечер), первая строка — дни недели (Пн–Вс). Каждая ячейка — кликабельный блок, при клике циклически переключает `slot_type` в порядке `empty → job → sql → eng → case → rest → empty`, сразу сохраняя через `update schedule_slots set slot_type=... where id=...`.
- Цвет ячейки зависит от `slot_type` (см. палитру в `00-overview.md`): job=amber, sql=teal, eng=blue, case=coral, rest=полупрозрачный серый, empty=пунктирная рамка.
- Легенда под сеткой: цветные точки + подписи всех 5 типов блоков.
- Текстовая подсказка под легендой: график — это шаблон, который можно сдвигать, если день не задался; важнее не пропускать дни подряд (привязка к streak-логике).

## Экран `/applications` — Отклики

**Назначение:** персональная CRM по вакансиям, на которые откликнулся пользователь.

**Данные:** `applications`, отсортированные по `created_at desc` (новые сверху).

**Компоненты:**
- Таблица с редактируемыми ячейками "inline" (без отдельной модалки редактирования — клик в ячейку = редактирование на месте, как в Google Sheets): колонки Компания, Вакансия, Источник, Дата отклика, Статус (select с цветным бейджем), Заметка, кнопка удаления строки.
- Изменение любого поля -> `update applications set [field]=... where id=...`, debounce ~500мс для текстовых полей, чтобы не делать запрос на каждый символ.
- Цвета статусов: `applied`=blue, `screen`=amber, `interview`=coral, `offer`=teal, `rejected`=приглушённый серый.
- Кнопка "+ Добавить отклик" сверху или снизу таблицы — создаёт новую строку с пустыми полями и `status='applied'`, `applied_date=сегодня`.
- На мобильной ширине экрана (если открыт веб с телефона) — таблица должна горизонтально скроллиться, а не ломаться.

**Пустое состояние:** если откликов нет — текст-приглашение добавить первый отклик.

## Экран `/progress` — Прогресс

**Назначение:** общая статистика + прогресс по обучающим трекам.

**Данные:**
- 4 карточки сверху: общее число `applications` (count), число `interview`+`offer` статусов (count), число `offer` (count), общее число выполненных `tasks` (count где `done=true`).
- Два прогресс-бара: `study_hours` где `track='eng'` (текущее/целевое значение, например «42 / 200 ч») и `track='sql'` (например «18 / 80 ч»).
- Под каждым прогресс-баром — кнопки `+1 час` / `-1 час`, инкрементирующие/декрементирующие `hours` в соответствующей строке `study_hours` напрямую через update.

**Визуал:** карточки-метрики в духе `metric card` (приглушённый фон, крупное число, мелкая подпись), прогресс-бар — простая горизонтальная полоса с заливкой соответствующим акцентным цветом трека (синий для eng, тиловый для sql).

## Экран `/resources` — Ресурсы и ссылки

**Назначение:** статический справочник — агрегаторы вакансий, стажировки, курсы. Не требует таблицы в БД, контент зафиксирован в коде (см. `05-resources-content.md` за полным списком и текстами).

**Компоненты:**
- Группы карточек по разделам: «Агрегаторы вакансий», «Стажировки», «SQL и аналитика», «Английский», «Подготовка к кейсам».
- Каждая карточка: заголовок, опциональный бейдж (бесплатно/платно/международные), список ссылок, короткая заметка серым текстом.
- Ссылки открываются в новой вкладке (`target="_blank" rel="noopener noreferrer"`).

## Экран `/login` — Вход

**Компоненты:**
- Простая форма email + пароль, переключатель "Войти" / "Зарегистрироваться".
- Через `supabase.auth.signInWithPassword` / `supabase.auth.signUp`.
- После успешной регистрации — вызвать сидирование данных нового пользователя (см. `01-data-model.md`, раздел "Сиды"), если оно не реализовано через БД-триггер.
- Опционально, если Cursor сочтёт это быстрее реализовать: вход через magic link (passwordless), это меньше кода на UI, но требует настройки email-провайдера в Supabase. Решение оставляем на реализацию — оба варианта приемлемы.

## Realtime-синхронизация

На каждом экране, где отображаются данные из Supabase, подписаться на изменения соответствующей таблицы:

```ts
const channel = supabase
  .channel('tasks-changes')
  .on('postgres_changes', { event: '*', schema: 'public', table: 'tasks', filter: `user_id=eq.${userId}` }, (payload) => {
    // обновить локальный state в соответствии с payload
  })
  .subscribe();
```

Это нужно, чтобы при изменении данных на телефоне веб-версия на компьютере обновилась без перезагрузки страницы (и наоборот). Не забыть `channel.unsubscribe()` при размонтировании компонента.
