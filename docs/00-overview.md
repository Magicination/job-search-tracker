# ТЗ: приложение-трекер «Поиск работы + обучение»

## Контекст и цель

Приложение для одного пользователя (Сергей, junior product manager), который сейчас:
- не работает (уволился, есть свободное время на полноценный поиск работы);
- ищет удалённую работу product manager-ом (junior/начинающий уровень);
- параллельно учится: подтягивает английский (B1 → B2) и SQL/продуктовую аналитику;
- хочет не терять мотивацию на длинной дистанции поиска работы (это главный риск — мотивация проседает без внешней структуры и видимого прогресса).

Цель приложения — быть персональным CRM + трекером привычек + календарём в одном месте, доступным с телефона и с компьютера, с синхронизацией данных между устройствами.

Это не SaaS и не публичный продукт. Об авторизации/мультипользовательности думаем только в объёме «один человек может зайти с двух устройств», без сложного шаринга, команд, ролей и т.д. Если что-то из стандартного боилерплейта Supabase auth даёт это бесплатно (email/password или magic link) — используем, но не строим лишнюю инфраструктуру под multi-tenant сценарии.

## Технологический стек

| Слой | Технология | Комментарий |
|---|---|---|
| Web-клиент | Next.js (App Router), TypeScript | Разворачивается на Vercel или аналоге |
| Mobile-клиент | React Native + Expo | iOS/Android, общая бизнес-логика максимально близка к web |
| Backend / БД | Supabase (Postgres + Auth + Realtime) | Synchronization из коробки через Supabase Realtime подписки |
| Стилизация web | Tailwind CSS | Тёмная тема по умолчанию (см. дизайн-токены ниже) |
| Стилизация mobile | NativeWind или styled-components | Та же палитра, что и в web |
| Графики / прогресс-бары | Recharts (web), Victory Native или react-native-svg (mobile) | Для прогресс-баров можно обойтись простыми View, без библиотек |
| Аутентификация | Supabase Auth, email+password или magic link | Простая форма логина, без соцсетей на старте |

## Архитектура верхнего уровня

```
┌─────────────────┐      ┌─────────────────┐
│   Next.js web    │      │  Expo mobile app │
│  (Vercel deploy) │      │  (iOS/Android)   │
└────────┬─────────┘      └────────┬─────────┘
         │                          │
         └──────────┬───────────────┘
                     │
              Supabase client SDK
                     │
         ┌───────────▼────────────┐
         │       Supabase          │
         │  - Postgres (данные)    │
         │  - Auth (email/pass)    │
         │  - Realtime (sync)      │
         │  - Row Level Security   │
         └─────────────────────────┘
```

Оба клиента (web и mobile) обращаются к одной и той же Supabase-базе напрямую через клиентский SDK (`@supabase/supabase-js`). Никакого отдельного REST/GraphQL backend-слоя писать не нужно — Supabase это закрывает.

Синхронизация между устройствами: подписка на изменения таблиц через Supabase Realtime (`supabase.channel(...).on('postgres_changes', ...)`), чтобы при изменении данных на одном устройстве другое устройство получало обновление без перезагрузки страницы.

## Структура проекта (рекомендация для Cursor)

```
/job-search-tracker
  /apps
    /web          — Next.js приложение
    /mobile       — Expo приложение
  /packages
    /shared       — общие типы (TypeScript), константы, бизнес-логика расчётов (streak, прогресс, бюджет часов), не зависящая от UI
    /supabase     — схема БД (SQL-миграции), сгенерированные типы Supabase
  /docs           — этот набор spec-файлов
```

Если монорепо со `/packages/shared` кажется избыточным для Cursor на старте — допустимо продублировать небольшую бизнес-логику (расчёт streak, расчёт % прогресс-бара) в обоих клиентах отдельно. Главное — одна и та же схема БД в Supabase для обоих клиентов.

## Дизайн-токены (общие для web и mobile)

Тёмная тема, вдохновлённая исходным HTML-прототипом:

```css
--bg: #14181f;
--panel: #1c222b;
--panel-2: #232a35;
--border: #2e3744;
--border-soft: #262e3a;
--text: #e7eaef;
--text-dim: #9aa5b3;
--text-faint: #677084;

--accent-amber: #e8a33d;   /* поиск работы / основной акцент / streak */
--accent-teal: #3fb88f;    /* обучение / SQL-аналитика */
--accent-blue: #5b8fd6;    /* английский */
--accent-coral: #dd6b4f;   /* кейсы/собеседования */
```

Шрифт: системный sans-serif стек (`-apple-system, "Segoe UI", Roboto, sans-serif`), без декоративных эффектов (без градиентов, теней, неона). Скругления 8–12px. Это рабочий инструмент, не маркетинговый лендинг — приоритет на читаемость и низкую визуальную нагрузку при ежедневном использовании.

## Нефункциональные требования

- Приложение должно открываться и быть пригодным к использованию с телефона (одной рукой, в первую очередь чек-лист на сегодня и быстрое добавление отклика).
- Любое действие (отметить задачу, добавить отклик, изменить график) должно сохраняться в Supabase немедленно (optimistic update в UI + запись в БД), без отдельной кнопки «Сохранить» — кроме текстового поля рефлексии дня, где сохранение по кнопке/блюру поля разумно.
- Оффлайн-режим не обязателен на первой итерации, но структура кода не должна делать его принципиально невозможным в будущем (например, не хардкодить прямые синхронные вызовы там, где позже потребуется очередь).
- Один пользователь = одна учётная запись Supabase Auth. Дополнительные пользователи не предусмотрены, но `user_id` должен присутствовать во всех таблицах с самого начала (на случай, если в будущем понадобится).
