# job-search-tracker

Трекер поиска работы + обучения. Полное ТЗ — в `/docs`.

> **Разворачиваете с нуля и не работали раньше с git/Vercel?** См.
> [`DEPLOY.md`](./DEPLOY.md) — там всё по шагам через GitHub Desktop, с
> картинками названий кнопок, без необходимости знать командную строку
> заранее. Разделы ниже — более компактная версия того же самого для тех,
> кто уже знаком с этими инструментами.

## Локальный запуск (web)

```bash
npm install
npm run dev
```

Это всё. Скрипт `scripts/check-env.js` сам:
- создаст `apps/web/.env.local` из шаблона, если его ещё нет;
- покажет инструкцию, если в файле остались значения-заглушки.

После того как заполните `.env.local` реальными ключами (см. ниже) и
выполните SQL-миграции — запустите `npm run dev` ещё раз, откроется
http://localhost:3000 с автоматическим редиректом на `/login`.

## Что за ключи и где их брать

Приложению (и локально, и на Vercel) нужны **два** значения из вашего
Supabase-проекта. Без них экраны откроются, но не смогут читать/писать данные.

### 1. Создайте проект в Supabase

Если у вас его еще нет — идите на https://supabase.com → "New project".
Бесплатного плана достаточно для личного использования. Дайте проекту любое
имя, выберите регион (например, ближайший к Тбилиси/Европе) и пароль для базы
(этот пароль от ключей ниже не зависит, он нужен для прямого доступа к
Postgres, в .env его вписывать не нужно).

### 2. Найдите Project URL и anon key

В дашборде вашего проекта откройте **Project Settings → API Keys**
(в боковом меню слева, иконка шестерёнки). Project URL можно также найти через
кнопку **Connect** в верхней части страницы проекта.

Там будет несколько значений — нужны именно эти два:

| Что ищете в дашборде | Куда вписать в `.env.local` |
|---|---|
| **Project URL** (вида `https://xxxxx.supabase.co`) | `NEXT_PUBLIC_SUPABASE_URL` |
| **anon key** / **anon public** (длинная строка вида `eyJ...`), в новых проектах может называться **publishable key** (`sb_publishable_...`) | `NEXT_PUBLIC_SUPABASE_ANON_KEY` |

**Важно:** на той же странице есть ещё **service_role key** (или **secret
key**) — его НЕ нужно использовать в этом проекте. Он даёт полный доступ к
базе в обход всех правил безопасности (RLS) и предназначен только для
серверного кода, который никогда не попадает в браузер. Приложение
полностью обходится anon-ключом — это безопасно благодаря RLS-политикам,
которые уже описаны в миграциях.

### 3. Впишите значения в `apps/web/.env.local`

```
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...
```

### 4. Настройте Supabase CLI и примените миграции

Раньше здесь была инструкция «скопируйте SQL в Dashboard → SQL Editor». Мы
от неё отказались: при ручном копировании легко забыть, что уже применяли,
а каждая новая фича добавляет ещё один файл, за которым руками следить
неудобно. Вместо этого — Supabase CLI (уже добавлен в зависимости проекта,
ставится при обычном `npm install`), который сам помнит, что уже применено
к вашей базе, и накатывает только новое.

Авторизация и привязка к вашему проекту (один раз на проект):

```bash
npx supabase login
npm run db:link
```

CLI спросит project-ref — это короткий идентификатор из URL вашего проекта
в Supabase Dashboard (`https://supabase.com/dashboard/project/XXXXXXX` — вот
этот `XXXXXXX`), либо предложит выбрать проект из интерактивного списка.

**Если вы уже создавали таблицы вручную через SQL Editor** (например,
следуя более ранней версии этой инструкции) — нужно один раз сообщить CLI,
что первые 3 миграции уже применены, без повторного выполнения их SQL:

```bash
npx supabase migration repair --status applied 20260101000001
npx supabase migration repair --status applied 20260101000002
npx supabase migration repair --status applied 20260101000003
```

Это только записывает отметку «применено» в служебную таблицу истории —
ничего не меняет в ваших данных и таблицах.

**Если у вас совсем новый проект без таблиц** — пропустите `migration
repair` и сразу выполните:

```bash
npm run db:push
```

Эта команда применит все файлы из `supabase/migrations/` по порядку. Среди
прочего одна из миграций создаёт приватный bucket `application-documents`
в Supabase Storage — туда попадают файлы резюме и сопроводительных писем,
которые пользователи прикладывают к версиям документов на странице
«Отклики». Это происходит автоматически, отдельно настраивать в Dashboard
ничего не нужно.

Дальше, по мере появления новых фич, цикл такой: создаёте новый файл
миграции (`npx supabase migration new <название>` создаст файл с
правильным timestamp в `supabase/migrations/`), пишете туда SQL (только
аддитивные изменения — `alter table add column`, никогда `drop table` или
`truncate`, чтобы не терять накопленные данные), затем `npx supabase db
push` — и изменение применится к реальной базе без необходимости что-либо
пересоздавать.

### 5. Локально готово

```bash
npm run dev
```

Зарегистрируйте аккаунт прямо на `/login` (email + пароль, кнопка
"Зарегистрироваться"). После регистрации триггер из шага 4 сразу заполнит
ваши первые данные (цели по часам, 3 стартовые задачи). Регистрация
открыта для всех — об ограничении доступа по приглашениям думать не нужно.

## Деплой на Vercel

Когда локально всё работает, выкладываем в продакшен. Supabase уже настроен
(шаги 1–4 выше) — для Vercel нужны те же два ключа, просто в другом месте.

### 1. Закиньте репозиторий на GitHub

```bash
git init
git add .
git commit -m "Initial commit"
```

Создайте пустой репозиторий на GitHub и запушьте туда (`git remote add
origin ...` → `git push -u origin main`). Vercel разворачивает именно из
git-репозитория, не из локальной папки — это даёт автодеплой при каждом
пуше.

### 2. Импортируйте проект в Vercel

На https://vercel.com → **Add New** → **Project** → выберите этот
репозиторий из списка. Это монорепо (несколько папок внутри одного
репозитория), поэтому **до первого деплоя** обязательно настройте:

- **Root Directory** → нажмите Edit и выберите `apps/web`
  (Vercel сам подтянет корневой `package.json`/`package-lock.json` и поймёт,
  что это npm workspace — Install Command трогать не нужно)
- **Framework Preset** → должен определиться автоматически как Next.js

Остальные настройки (Build Command, Output Directory) можно оставить по
умолчанию.

### 3. Добавьте переменные окружения

В том же экране импорта (или позже в **Project Settings → Environment
Variables**) добавьте те же два значения, что и в `.env.local`:

| Key | Value |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | ваш Project URL из Supabase |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | ваш anon/publishable key из Supabase |

Важно: переменные с префиксом `NEXT_PUBLIC_` Next.js "запекает" в код прямо
на этапе сборки. Если добавить или изменить их после деплоя — нужен новый
деплой (Vercel → Deployments → ⋯ → Redeploy), просто перезапуск приложения
их не подхватит.

### 4. Deploy

Жмите **Deploy**. Через 1-2 минуты Vercel выдаст рабочий URL вида
`your-project.vercel.app`. Дальше каждый `git push` в основную ветку будет
автоматически пересобирать и обновлять прод.

### Если поменяется домен или CORS

Supabase Auth по умолчанию не блокирует запросы по домену для самого
клиента (это не CORS-ограниченный сценарий, как для серверных secret-ключей),
так что anon key с любого домена будет работать после деплоя без
дополнительной настройки. Если захотите настроить magic-link письма или
redirect-URLs — это делается в Supabase → Authentication → URL
Configuration, добавив туда ваш `*.vercel.app` домен.

## Структура репозитория

```
/apps
  /web          — Next.js приложение
/packages
  /shared       — общие типы и бизнес-логика (streak, прогресс обучения, аналитика откликов)
/supabase
  /migrations   — версионированные SQL-миграции (управляются Supabase CLI)
  config.toml   — конфигурация CLI
/docs           — исходное ТЗ (6 файлов)
/scripts        — вспомогательные скрипты (check-env.js)
```

## Как добавлять новые фичи в базу данных

Любое изменение схемы — это новый файл миграции, никогда правка существующих
или ручные правки в Dashboard:

```bash
npm run db:new -- добавить_что-то
# → создаёт supabase/migrations/<timestamp>_добавить_что-то.sql
```

Откройте созданный файл, напишите SQL (новые таблицы, `alter table add
column`, новые индексы — всегда аддитивно, без `drop`/`truncate` на
существующих данных), затем:

```bash
npm run db:push
```

Если меняете и TypeScript-типы в `@job-search-tracker/shared` (см.
`packages/shared/src/types.ts`) — обновите их в той же сессии работы, чтобы
типы не расходились со схемой.
